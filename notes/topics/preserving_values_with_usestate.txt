If we break it down, we've seen how you create your View with JSX, and how you encapsulate that View inside of a function to get your component. But there's still one piece of the formula we need to talk about, state.

State, and specifically the ability for individual components to own and manage their own state, is what makes React so powerful, and it's what allows you to build complex user interfaces out of simple, isolated components.

So what does adding state to a component look like? Let's say we were building a simple counter app that keeps track of how many times a user has clicked a button.

Your first assumption might be to do something like this.

    export default function Counter() {
        let count = 0
        const handleClick = () => count++
        return (
            <button onClick={handleClick}>
            {count}
            </button>
        )
    }    

After all, React is just JavaScript, right? And the way you add state to a function is by creating a variable. Unfortunately, this won't work. Can you spot why?

Ironically, it won't work for the same reason, because React is just JavaScript – and that's not how JavaScript works.

This is clearer if we take React out of the picture entirely. Here's a normal JavaScript function.

    function counter() {
        let count = 0
        count++
        return count
    }

    console.log(counter())
    console.log(counter()) 
    console.log(counter()) 

my response: because count is a local variable that will be reset every time you call the function 

Without React, you wouldn't expect the count variable to persist across different function invocations. Each invocation is its own isolated environment with its own scope and context. With React, it's no different.

So how do we get around this?

Well, we need some help from React. Specifically, we need a way to tell React to preserve the value of count between different invocations of our Counter function (React refers to these function component invocations as renders).

Lucky for us, React comes with a hook that does exactly this called useState.

    Hook 'Em

        useState is the first of many hooks that you'll learn in this course. You can think of hooks as special functions that allow you to get help from React.

        The reason they're special is because they have a few limitations. Namely that they must be prepended with use and you can’t call hooks inside conditions, loops, or other nested functions.

        We'll go deeper into hooks throughout the course, but here's a simple mental model you can use for them.

        "Hooks are functions, but it’s helpful to think of them as unconditional declarations about your component’s needs. You use React features at the top of your component similar to how you import modules at the top of your file." - React docs

useState

    useState is the blessed way to preserve a value across component renders. It comes built-in with React and can be accessed via React.useState. It takes in a single argument, the initial value for that piece of state, and returns an array with the first item being the state value and the second item being a way to update that state.

    const stateArray = React.useState("initial state value")
    const state = stateArray[0]
    const setState = stateArray[1]

    console.log(state) // "initial state value"
    setState('new state value')

    Array Destructuring

        You may notice that the code above feels a little clunky. This stems from the fact that useState returns an array. We can clean this up by using array destructuring.

        If you're not familiar, array destructuring allows you to pluck multiple items from an array and assign them to variables in bulk.

            const scores = [86, 91, 82]
            const [firstGame, secondGame, thirdGame] = scores

        Because useState returns an array, we can use array destructuring to clean up our useState invocation by assigning the first item to state and the second item to setState.
